# desktop / Screen_Mails

## Purpose
Screen_Mails is not a user-facing screen at all — it has no navigation entry point, no visible layout, and zero SVG assets (confirmed: no svg/desktop/Screen_Mails folder exists, unlike every other desktop screen). It is a hidden "email template host": a parking screen holding HtmlViewer controls whose HtmlText property builds the full HTML body of transactional emails. Other screens (Screen_Compras, Screen_Aprobaciones, Screen_OrdenesTrabajo) read these controls' .HtmlText at runtime and pass it into Office365Outlook.SendEmailV2 as the email body. This is the classic Power Apps "hidden email-template screen" pattern used because Power Fx has no native string-templating/HTML-builder outside a control's Text property.

## Layout
No visual layout — this screen is never navigated to by the user. It contains exactly 6 non-visual/off-screen children, positioned arbitrarily (X/Y overlapping, e.g. two controls both at X=875) since position is irrelevant: 1) HtmlTEnviarAprobacion (HtmlViewer, W522 H474) — "compra enviada a aprobación" template. 2) lbl_correos_SM (Text, Visible=false, X40 Y40) — a hidden label whose .Text is a hardcoded semicolon-separated BCC address list (internal dev/admin emails), reused as the Bcc parameter on nearly every SendEmailV2 call app-wide. 3) html_CompraAprobadaGerencia (HtmlViewer, W491 H344 X875) — "compra aprobada" template. 4) html_ordenResuelta (HtmlViewer, W491 H375 X875 Y393) — "OT resuelta" template. 5) html_CompraRecibida (HtmlViewer, W734 H310 X632) — "compra recibida" template. 6) html_CompraAprobadaSupervision (HtmlViewer, W632 H300 Y468) — "compra aprobada por supervisión" template; verified DEAD CODE, referenced only inside a commented-out block in Screen_Aprobaciones (line 473), never actually invoked by any active SendEmailV2 call.

## components
- HtmlTEnviarAprobacion (HtmlViewer) — renders subject 'Orden de Compra Enviada a Aprobación': header, order number (InfoAprobacion.ID), a table iterating CollectResumenCompra (Edificio_DC, Articulo_DC, Cantidad_DC, CostoUnitario_DC formatted as currency, CostoTotal_DC), a computed grand total via Sum(CollectResumenCompra, CostoTotal_DC), and a signature line with VarUser.UsuarioApp_Usr. Table header styled #23313E bg / white text.
- lbl_correos_SM (hidden Text control) — not a UI component, functions as a shared 'constant' holding the BCC distribution list string; a config value smuggled into a control property because Power Fx has no global string constant outside a Set()/named formula in this app's era.
- html_CompraAprobadaGerencia (HtmlViewer) — subject 'Orden de Compra Aprobada': same table shape as above but sourced from CollectAprobacionesMail, signed with VarUser.UsuarioApp_Usr.
- html_ordenResuelta (HtmlViewer) — subject 'Resolución - Orden de Trabajo Nro {RegistroOT.ID}': narrative paragraph naming the asset (RegistroOT.ConcatActivo_IN), a conditional 'Repuestos Utilizados' table (only rendered If CountRows(CollectRepuestosFinalizacion)>0, else a 'Sin repuestos utilizados' line), work metadata (TipoTrabajo_IN, TipoTarea_IN, DiasEstimado_IN) and a computed 'Días utilizados' via DateDiff(DateValue(RegistroOT.FechaAsignada_IN), Today()), signed by RegistroOT.Tecnico_IN.
- html_CompraRecibida (HtmlViewer) — subject 'Orden de Compra Recibida': order number InfoRecibir.ID, a 7-column receiving table over Filter(CollectRecibirCompra, Status_DC='Activo') (Edificio, Articulo, Cantidad Pedida, Cantidad Recibida with '-' fallback via IsBlank(Recibido_DC), Precio Unitario, Precio Total Pedido, and a COMPUTED Precio Total Recibido = CostoUnitario_DC * Recibido_DC done inline in the HTML string), plus reception notes (txt_obsRecepcion.Value with '-' fallback), two Sum() totals (ordered vs received), signed VarUser.ConcatName_Usr.
- html_CompraAprobadaSupervision (HtmlViewer) — DEAD template, same shape as html_CompraAprobadaGerencia, subject would be 'Orden de Compra Aprobada Por Supervisión', signed VarUser.ConcatName_Usr; only reachable from commented-out code, not part of the live app.

## modals

## data_reads
- CollectResumenCompra (in-memory collection, populated elsewhere in Screen_Compras when building a purchase order) — read by HtmlTEnviarAprobacion for Edificio_DC/Articulo_DC/Cantidad_DC/CostoUnitario_DC/CostoTotal_DC and Sum(...,CostoTotal_DC).
- CollectAprobacionesMail — read by html_CompraAprobadaGerencia and the dead html_CompraAprobadaSupervision for the same 5 columns.
- CollectRepuestosFinalizacion — read by html_ordenResuelta for Repuesto_ROT/Cantidad_ROT, gated by CountRows(...)>0.
- CollectRecibirCompra — read by html_CompraRecibida via Filter(CollectRecibirCompra, Status_DC = 'Activo') for the receiving table and both Sum() totals.
- InfoAprobacion / InfoAprobada / InfoRecibir / RegistroOT (in-memory record variables, one per source screen, set via Set() before the email is triggered) — read for order/OT IDs and metadata.
- VarUser (global user-session record) — VarUser.UsuarioApp_Usr and VarUser.ConcatName_Usr used as email signatures / to gate 'Admin' branch logic in callers.
- None of the 5 SharePoint lists ('14.Compras', '15.DetalleCompras', '16.Aprobaciones', OT list, etc.) are read directly on this screen — all data arrives pre-shaped into collections/records by the calling screens (Screen_Compras, Screen_Aprobaciones, Screen_OrdenesTrabajo) before SendEmailV2 fires.

## data_writes
- This screen performs NO Patch/Remove/ClearCollect writes itself — it is purely a read+render template layer. All SharePoint writes ('14.Compras' Status_C, '16.Aprobaciones' Status_AP, '08.Stock', '08.MovimientoStock', '15.DetalleCompras') happen in the CALLING screens' OnSelect handlers, immediately before or after the Office365Outlook.SendEmailV2 call that consumes this screen's HtmlText.
- The only 'send' action attributable to this screen's content: Office365Outlook.SendEmailV2 calls made from Screen_Compras.pa.yaml (bt 'aceptar' handlers) and Screen_Aprobaciones.pa.yaml (bt_aprobarAP) and Screen_OrdenesTrabajo.pa.yaml, all following the pattern If(VarUser.UsuarioApp_Usr = 'Admin', SendEmail(to: hardcoded internal admin address, ..., Bcc: lbl_correos_SM.Text), SendEmail(to: LookUp(CollectEmailCompra/CollectMails, Modulo_E = <module>, EmailConcat_E), ..., Bcc: lbl_correos_SM.Text)) — i.e. admins get a direct copy, everyone else's mail is BCC'd to the fixed internal dev/admin distribution list from lbl_correos_SM.

## navigation
- None. No Navigate() targets this screen and this screen contains no Navigate() calls itself — it is reached only implicitly, by other screens referencing its child controls' properties (e.g. HtmlTEnviarAprobacion.HtmlText) without ever making it the visible screen.

## statuses
- Status_C: 'Pendiente', 'Aprobacion', 'Aprobada', 'Recibida' (Compras list, referenced by callers of these templates)
- Status_AP: 'Pendiente', 'Aprobada Supervision', 'Aprobada' (Aprobaciones list)
- Status_DC: 'Activo', 'No Recibido' (DetalleCompras — used to filter which lines appear in the 'Compra Recibida' email table)

## role_logic
No direct role/permission check happens on Screen_Mails itself (it has no OnVisible, no CollectPermisos reference). The gating lives entirely in the CALLING screens: every SendEmailV2 invocation that uses one of these templates is wrapped in If(VarUser.UsuarioApp_Usr = \"Admin\", <send to fixed internal admin address>, <send to LookUp(CollectMails/CollectEmailCompra, Modulo_E = \"<module>\", EmailConcat_E)>) — i.e. the recipient (not the visibility of the template) branches on whether the acting user's profile is 'Admin'. VarUser.Perfil_Usr is also used just after the html_CompraAprobadaGerencia send in Screen_Aprobaciones to decide which Status_AP filter to reload into CollectAprobaciones ('Gerencia' vs 'Admin' vs default), but that's post-send list-refresh logic, unrelated to this screen's templates. No 99.ABM_ListaPermisosPerfilesV3 reference appears in this screen or in the three callers' snippets reviewed for these specific triggers."]

## visual_notes
No SVGs exist for this screen (verified: svg/desktop/ has no Screen_Mails subfolder, unlike all 9 other desktop screens which do). All visual styling is inline CSS inside the generated HTML strings, consistently: table header background #23313E (dark navy, matches the app's primary brand color RGBA(56,96,178) family) with white (#FFFFFF) text; body text #000000, font-family Arial/sans-serif, 14px; row separators '1px solid #CCCCCC'; headings use emoji prefixes (📤 outbound/approval, 📥 received, 🛠 work-order resolution, 📋 detail, 💰 total, 🧾 type, 🔧 task, 📆/⏳ dates, 📌 notes) as a lightweight visual/semantic marker in place of icons. All currency formatted via Text(Value(x), \"##,##00.00\") with a literal '$ ' prefix — no locale-aware Intl formatting, no thousands-separator distinct from decimal handling (risk: '##,##00.00' is a fragile hand-rolled mask, not a proper currency format)."]

## react_mapping
Do not port this screen as a "screen" at all — it has no route, no visible UI, and porting it as one would be pure over-engineering. Instead, replace the whole pattern with actual code: (1) Build small server-side (or shared) email-template functions/components — one per template (PurchaseSentForApprovalEmail, PurchaseApprovedEmail, WorkOrderResolvedEmail, PurchaseReceivedEmail) — using a proper templating approach (React Email / JSX-to-HTML, or a plain template-literal function) instead of Power Fx string concatenation; each takes typed data (order lines, totals, user) as props/args instead of reading ambient collections/records. (2) Replace lbl_correos_SM with a real constant/env var (e.g. NOTIFICATIONS_BCC_LIST in config), not a hidden UI control. (3) Replace the hardcoded 'santiago.bianucci@...' / 'mantenimiento@thetoprentals.com' addresses and the VarUser.UsuarioApp_Usr === 'Admin' branch with a proper notification-recipient resolver (role/module → recipient list) driven by a Supabase table (mirroring CollectMails/CollectEmailCompra's Modulo_E → EmailConcat_E lookup) rather than inline conditionals duplicated across 3 screens. (4) Fix the currency formatting to Intl.NumberFormat('es-AR', {style:'currency', currency:'ARS'}) instead of the fragile '##,##00.00' mask, and compute derived values (CostoUnitario_DC * Recibido_DC, Sum() totals) server-side instead of re-deriving them in the template string. (5) Delete html_CompraAprobadaSupervision entirely — it is confirmed dead code with no live caller; do not port it. (6) Trigger sends from a backend function (Supabase Edge Function / API route) called after the Patch/status-update transaction commits, not from client-side UI code — this also fixes the current architecture flaw where email delivery is coupled to a Button.OnSelect and silently fails if the user closes the tab mid-handler. No Sumar UI Kit primitives apply here since there is no UI to render; the only 'kit' touchpoint is a Toast confirming 'Email sent' / 'Purchase order marked as received' on the calling screens after the backend call resolves.

# desktop / Screen_Aprobaciones

## Purpose
Back-office purchase-approval workspace. Shows purchase requests waiting on approval (list backed by SharePoint list '16.Aprobaciones', linked to '14.Compras'/'15.DetalleCompras') and lets Gerencia/Admin/Compras users inspect line-item detail, approve, reject (with a reason), or edit a still-Pendiente request's items before it moves forward. Visibility of the queue itself is role-scoped: Gerencia only sees requests already "Aprobada Supervision"; Admin sees "Aprobada Supervision" + "Pendiente"; every other role sees only "Pendiente". Approving sends a notification email; rejecting soft-cancels the purchase and its detail lines.

## Layout
Fixed left sidebar (Group GroupExtras_AP, ~150px, dark navy #23313E per background SVG): app logo, vertical icon nav gallery gal_menu_HM_4 (role-filtered), logout icon pinned near bottom. Main area is a big white rounded card (per BackgroundImage SVG 001) starting at X=163: a top bar inside the card holds the search input (txt_search_AP, top-right), a refresh icon button, and a filter icon button that toggles a small filter popover panel (month + status multiselects + "Filtrar" button). Below the top bar sits the primary content: gal_aprobaciones, a vertical row-based gallery (TemplateSize 52, X=183 Y=124 W=1150) acting as the "table" of approval requests, each row rendering a status badge, several text columns, and up to 4 action icons (view/reject/edit/approve) gated by status+role. An empty-state illustration (checkmark circle + "sin resultados" copy) shows when the collection is empty. Six full-screen (1366x768) dimmed-overlay Groups implement modals: reject, approve, view-detail, edit-quantity, add/edit-purchase, and logout-confirm — plus a full-screen loading spinner overlay bound to LoadingLocal.

## components
- Sidebar nav — logo + vertical icon gallery (gal_menu_HM_4) driven by CollectPermisos filtered on VarUser.Perfil_Usr (Admin/Operador/Tecnico/Recepcion; Compras profile gets no menu at all — Blank()), sorted by field_8; each item Switches on Modulo_LPP to prefetch collections and Navigate to the target screen.
- Search input (txt_search_AP) — free-text filter matched against Status_AP or IDCompra_A on the approvals gallery, purely client-side/reactive (no explicit OnSelect handler).
- Refresh icon button (refresh_AP) — re-Refreshes '16.Aprobaciones' and re-runs the role-scoped ClearCollect(CollectAprobaciones, ...) exactly like initial load.
- Filter toggle + popover (Group_Filtros_AP / bt_popUpFiltrar_AP / img_filtro_AP) — month multiselect (cmbox_mes_AP, last 12 months, defaults to current month) and status multiselect (cmbox_estado_AP: Aprobada/Aprobada Supervision/Rechazada/Pendiente, defaults 'Todos'), applied via bt_filtrar_AP which re-ClearCollects CollectAprobaciones by FechaMesAñoGen_AP then narrows by Status_AP.
- Approvals gallery (gal_aprobaciones) — main list/'table': columns Fecha_AP, Sector_AP, Urgencia_AP, Tecnico_AP, 'N° '+IDCompra_A, UserGen_AP, Cantidad_AP, formatted Monto_AP ($ es-ES), and a status badge image (6 variants). Sorted by Fecha_AP desc, or by Or(search in Status_AP, search in IDCompra_A) when search box has text. Visible only when CollectAprobaciones has rows; otherwise the empty-state illustration shows.
- Row action: view-detail icon (img_detailAP) — opens ViewDetailCompra_AP popup with that purchase's active (or inactive, if the request was rejected) detail lines plus its Fecha_C/Observacion_C looked up from '14.Compras'.
- Row action: reject icon (img_rechazarAP) — visible only when Status_AP in {Pendiente, Aprobada Supervision} and role in {Gerencia, Admin}; opens the reject modal.
- Row action: edit icon (img_editAP) — visible only when Status_AP = 'Pendiente'; opens the add/edit-purchase modal pre-loaded with that request's items; its X position shifts depending on role (extra room reserved when Gerencia/Admin, who also see the reject icon in that row).
- Row action: approve icon (img_aceptarAP) — visible only when Status_AP in {Pendiente, Aprobada Supervision} and role in {Gerencia, Admin, Compras}; opens the approve confirmation modal.
- Empty-state illustration (warning_AP) — checkmark-in-circle art with 'no pending items' copy, shown when the gallery is empty.
- Full-screen loading overlay (Loading_AP) bound to context var LoadingLocal, wrapped around essentially every async block on the screen.

## modals
- Reject modal (Group_Rechazar_AP / PopUpRechazarCompra) — full-screen dim overlay + white card; free-text reason field (info_Rechazo); 'Guardar' patches Aprobaciones→Rechazada+ObsRechazo_AP, Compras→Rechazada, soft-deletes all active detail lines, then re-fetches the role-scoped queue; 'Cancelar' just resets the field and closes.
- Approve modal (Group_Aprobar_AP / PopUpAprobarCompra) — confirmation card; 'Aceptar' patches Compras→Aprobada and Aprobaciones→Aprobada with approval metadata (date/month/year/hour/user), sends the approval email, re-fetches the queue; 'Cancelar' just closes. Sets/clears a global InfoAprobada record around the action (consumed by the off-screen email template).
- View-detail modal (ViewDetailCompra_AP / PopUpViewDetail) — read-only: header date (FechaCompra) + observations (ObsCompra, falls back to 'No hay observación...'), and a gallery of purchase line items (quantity badge + article name) sourced from active or inactive detail lines depending on whether the request was rejected. Two close affordances (X and bottom bar).
- Edit-quantity modal (Group_EditCantidadItem_AP / PopUpEditCant) — nested modal opened from inside the add/edit-purchase modal; single numeric input (txt_cantidadItem_1) seeded from CantidadEdit; 'Aceptar' recalculates CostoTotal_DC = qty*unitCost and patches the local CollectCompraEdit only (no SharePoint write yet); 'Cancelar'/X discard.
- Add/edit-purchase modal (GroupAddCompra_AP / PopNuevaCompra || PopUpEdit) — the heaviest popup, reused to edit a Pendiente approval's underlying purchase: user combobox, urgency combobox (Baja/Media/Alta), building + article comboboxes with quantity input and an 'add line' button that appends to a local CollectCompraEdit, observations textarea, and a line-items gallery (each row editable via the edit-quantity modal or soft-deletable). 'Aceptar' (disabled until user+urgency selected and at least one line exists) performs the full Aprobaciones/Compras/DetalleCompras write sequence described above; 'Cancelar'/X clear everything without saving.
- Filter popover (Group_Filtros_AP / Filtros) — month multiselect + status multiselect + 'Filtrar' button that re-slices CollectAprobaciones client-side; a small X collapses it back to the filter icon.
- Logout confirmation (Group_CerrarSesion_AP / cerrarSesion) — standard cross-screen pattern: 'Aceptar' resets login fields and Navigates to Screen_Login; 'Cancelar'/X just closes.
- Loading overlay (Loading_AP / LoadingLocal) — full-screen spinner, not a decision modal but toggled around every write/fetch block.

## data_reads
- '16.Aprobaciones' — Refresh() then role-gated Filter into CollectAprobaciones: Gerencia -> Status_AP='Aprobada Supervision'; Admin -> Status_AP='Aprobada Supervision' Or 'Pendiente'; all other roles -> Status_AP='Pendiente'. Re-run identically on screen load, on refresh click, and after every approve/reject/save mutation.
- '14.Compras' — LookUp(ID = Value(ThisItem.IDCompra_AP)) to fetch Fecha_C/Observacion_C for the detail popup, and to fetch the full purchase record (RegistroCompra) when opening the edit-purchase modal.
- '15.DetalleCompras' — Filter(IDCompra_DC = ThisItem.IDUnivoco_AP And Status_DC = 'Activo' [or 'Inactivo' if the request was Rechazada]) for the read-only detail viewer; Filter(Status_DC='Activo' And IDCompra_DC=...) into CollectCompraEdit when opening the edit modal; also read for the approval-email line list (CollectAprobacionesMail).
- '99.ABM_Emails' — ClearCollect(CollectEmailCompra, '99.ABM_Emails') then LookUp(Modulo_E='Aprobaciones') to resolve the notification recipient when the approving user isn't Admin.
- CollectPermisos (role/menu permission cache, presumably sourced from '99.ABM_ListaPermisosPerfilesV3' at login) — filtered by Admin_LPP/Operador_LPP/Tecnico_LPP/Recepcion_LPP='SI' to build the side nav.
- CollectEdificios / CollectArticulos / CollectUserABM (buildings, article catalog, users) — pre-existing client collections consumed as Items sources for the edit-purchase modal's comboboxes; not (re)loaded by this screen.
- Shared side-nav Switch also prefetches other screens' lists on navigation ('07.OrdenesTrabajo', '08.Stock', '99.ABM_Edificios', '99.ABM_TipoUnidades', '19.Ventilaciones', '00.Usuarios', '99.ABM_Frecuencias', '99.ABM_Articulos') — not read for this screen's own logic, just cache warms for the target screen.

## data_writes
- Patch '16.Aprobaciones' — reject: {Status_AP:'Rechazada', ObsRechazo_AP: <reason text>}. approve: {Status_AP:'Aprobada', FechaAprobada_AP, FechaMesAñoAprob_AP, FechaAñoAprob_AP, Hora_AP, User_AP:VarUser.UsuarioApp_Usr}. edit-save: {Monto_AP: Sum(active detail lines, CostoTotal_DC), Cantidad_AP: Sum(active detail lines, Cantidad_DC)}.
- Patch '14.Compras' — reject: {Status_C:'Rechazada'}. approve: {Status_C:'Aprobada'}. edit-save: {Título:'sumar', UsuarioCompra_C, UrgenciaPedido_C, Observacion_C, CantidadTotal_C, MontoTotal_C}.
- Patch '15.DetalleCompras' — reject: ForAll matching active lines -> {Status_DC:'Inactivo'} (soft-delete, no Remove() ever called). edit-save: ForAll existing lines -> {Status_DC, Cantidad_DC, CostoTotal_DC}; ForAll new (ID blank) lines -> Patch(Defaults('15.DetalleCompras'), {...}) as an insert-via-Patch pattern, always with Status_DC:'Activo'.
- Patch(CollectCompraEdit, ...) — local-only, client-side collection edits (line-item quantity change recalculating CostoTotal_DC = qty*unitCost; line soft-delete to Status_DC:'Inactivo') before the user hits Save; nothing hits SharePoint until bt_AceptarCompra_SC_1.
- Office365Outlook.SendEmailV2 — fires once on approve, subject 'Compra - Aprobada', HTML body from html_CompraAprobadaGerencia.HtmlText (a control not defined on this screen — cross-screen shared template), Bcc lbl_correos_SM.Text; recipient is a hardcoded address if VarUser.UsuarioApp_Usr='Admin', otherwise the '99.ABM_Emails' lookup for module 'Aprobaciones'. A second, larger email block for a 'Compra - Pendiente de Aprobación' / supervision-stage notification is present but fully commented out (dead/legacy code, not executed).

## statuses
- Status_AP: Pendiente, Aprobada Supervision, Aprobada, Rechazada, En Aprobacion, Recibida (all 6 have dedicated badge art, but only the first 4 appear in the filter combobox and in any write path on this screen — 'En Aprobacion' and 'Recibida' must be set by a different, not-yet-analyzed screen, e.g. goods receipt).
- Status_C ('14.Compras'): Pendiente, Aprobada, Aprobacion, Rechazada.
- Status_DC ('15.DetalleCompras'): Activo, Inactivo (soft-delete flag; Remove() is never called anywhere on this screen).
- Urgencia_AP / UrgenciaPedido_C: Baja, Media, Alta.
- Perfil_Usr (role) values referenced: Admin, Operador, Tecnico, Recepcion, Compras, Gerencia.
- Filter combobox sentinel: 'Todos' (means no status filter applied).

## visual_notes
Card-on-dark-canvas layout: outer screen fill is solid black/navy, main content is a white rounded-corner (19px) card (#FFFFFF, border #E3E8ED) offset from a dark sidebar strip (#23313E). Search box and filter button are light pill inputs (#F1F5F8 / #FEFEFE fill, #DFE6EB border). Table header row background is a light steel-blue tint (#E3E9EE) with grey icon-label text (#6B7680); a search-glass and filter-funnel icon sit in the header pills. Status badges are soft-tinted rounded-rect chips, each with a matching darker text/border tone: Pendiente = amber (#FFE7C2 fill / #FEC877 border / #C07402 text), Rechazada = dusty red (#E2B1B1 / #865959 / #865959 text), Aprobada = mint green (#DBFFCA / #CBEABD / #3A8517 text), En Aprobacion = pale blue (#ECF9FF / #9BD0EB / #0594DD text), Recibida = pale blue narrow variant (#EDF9FF / #CAE7F5 / #0594DD text), Aprobada Supervision = mint green wide variant (same green family, wider chip to fit the longer label). Row action icons: eye (view, stroke #23313E), red X-in-rounded-square (reject, #F6D7D0 fill/#D9715B border/#884335 glyph), pencil-with-slash-line style icon (edit), green check-in-rounded-square (approve, #E2FFE0 fill/#ACDEA9 border/#63B75C glyph). Empty-state illustration: a big grey circle with a bold dark checkmark plus explanatory copy, card style matching the rest of the UI (white bg, #C8D1D9 border). Modal chrome throughout (reject/filter panels) is a plain white rounded card with a light-grey input pill (#F5F5F5/#D8D4D4) and a solid near-black CTA button (#232323/#363636) with white label text — a consistently dark 'primary button' treatment distinct from the airy content areas. Overall palette is a cool grey/blue/white back-office aesthetic with semantic status colors (amber/red/green/blue) doing all the color signaling.

## react_mapping
Sidebar layout primitive for the persistent left nav (icon-only, active state = current Modulo_LPP) + logout affordance pinned bottom, gated by a `usePermissions(profile)` hook mirroring the CollectPermisos filter (including the 'Compras profile gets no nav' edge case). Header row: Input (search, with clear icon) + IconButton (refresh, spinner-aware) + a Popover (not a full Drawer — content is just 2 selects + 1 button) anchored to a filter IconButton, containing two Combobox/multi-select controls (month range, status) and a primary Button 'Filtrar'. Main list: Table (not a raw gallery) with columns Fecha, N° Compra, Usuario, Sector, Urgencia, Técnico, Cantidad, Monto (MoneyInput-formatted, read-only cell), Estado (StatusBadge with a variant map: pendiente=warning, rechazada=danger, aprobada=success, 'aprobada-supervision'=success-wide, en-aprobacion=info, recibida=info), and a row-actions cell of icon Buttons wrapped in Tooltip, each individually visibility-gated by (status ∈ set) && role check — port that gating as a small pure function, not scattered JSX conditionals, since the row currently juggles 3 independent Or/And rules. Empty state: Card + illustration + copy, shown when the filtered/queried rows are empty (React Query's `isEmpty` rather than a client-side CountRows check). Reject/Approve: Modal recipe (confirm-dialog variant) — Reject needs a required Textarea (reason) before its confirm Button enables; Approve is a plain two-button confirm; both should surface a Toast on success/failure (PowerApps gives none today — a UX gap worth closing) and should NOT block the email send on the client thread — model it as a fire-and-forget server action or edge function once emails move off Office365Outlook. View-detail: Modal (read-only variant) rendering a small Table/list of line items plus two text fields (date, observation) — good fit for a simple non-editable Card list, no need for a full DataTable. Edit-quantity: this is really just an inline row-editor; skip the nested-modal ceremony from Power Apps and do it as an inline editable cell or a tiny inline Popover on the line-item row instead of stacking a second full-screen overlay on top of the purchase-editor modal. Add/edit-purchase: the biggest piece — Modal (large/form variant) composing Combobox (user, article — searchable, since article list can be long), Select (urgency, building), a bare Input/NumberInput for quantity, MoneyInput-driven cost display (unit cost from the article catalog, computed total), a 'add line' Button, and an embedded Table of pending line items with per-row edit/remove — this is effectively a mini order-builder and should own its own local form state (React Hook Form / Zod) mirroring the Power Apps CollectCompraEdit local-buffer-then-commit pattern (don't touch Supabase until final Save). Logout confirm: reuse the shared AppShell's confirm-Modal, this is identical across every screen in the app, not screen-specific. Loading overlay: replace with per-action Button loading state + Table-level skeleton rather than a screen-wide spinner — the PowerApps LoadingLocal blocks the whole UI for even trivial reads, which is the main friction point to fix. Flag for backend migration: the SharePoint list has no real hard-delete anywhere in this flow (only Status_DC/Status_AP flag flips) — decide deliberately in Supabase whether to keep soft-delete semantics or move to real relational deletes+FKs; also the approve email recipient logic keys off VarUser.UsuarioApp_Usr==='Admin' while every other role check in the app uses Perfil_Usr — that inconsistency should be resolved (not silently ported) when rebuilding the notification rule in the new backend. Mobile: table rows become stacked Cards (status badge top-right, actions as a bottom action-row or overflow menu), and the purchase-editor Modal should switch to a bottom-sheet on small viewports given its length.

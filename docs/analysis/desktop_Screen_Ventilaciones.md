# desktop / Screen_Ventilaciones

## Purpose
Back-office screen for managing "Ventilaciones" — the recurring air-duct/ventilation cleaning maintenance task per building unit (room). Admins/supervisors create a recurring cleaning schedule for a unit, assign it to a technician with a due date, track its status through a small workflow (Pendiente → Asignada → Programada → Realizada, or Eliminada), and delete pending ones. It is the desktop counterpart to whatever the mobile app uses for technicians to mark cleanings done (that write path is not in this screen).

## Layout
Shared app shell (background 001.svg): dark navy (#23313E) outer frame, white rounded content card (1190x732) inset at X=162,Y=17.7. Left column (X=5..148) holds the app logo (img_iconoApp_5), a vertical role-filtered nav gallery (gal_menu_HM_5, TemplateSize 44) and a logout image button pinned near the bottom. Main content area (X≈185+) has: a toolbar row (~Y=38-42) with an "Agregar Edificio" button, a search TextInput, a refresh icon, and a filter trigger button anchored top-right; below it (Y=144) the main data gallery gal_ventilaciones (BrowseLayout two-text-one-image variant, TemplateSize 49, width 1145) listing rows with a status badge, building, room, last-cleaning date, next/scheduled date and row action icons. An empty-state illustration overlays the list area when it has zero items. Six full-screen (1366x768) dark-scrim overlay popups (Group_* / Filtro* groups, each an Image containing a white rounded card at a fixed X/Y) implement: filter panel, logout confirm, delete confirm, create-ventilación form, assign-ventilación form (two size variants), and a "view detail" panel for completed items — plus a full-screen loading spinner overlay gated on LoadingLocal.

## modals

## data_reads
- '19.Ventilaciones' — main list. Refresh() then ClearCollect(CollectVentilaciones, Filter(..., Estado_VE="Pendiente" Or "Asignada" Or "Programada")) on screen entry/refresh/nav-in from sidebar; re-filtered client-side in the gallery by DateDiff(Today, ProximaLimpieza_VE)<=90 and by search text over Edificio_VE/Habitacion_VE/Estado_VE
- '99.ABM_TipoUnidades' — building units/rooms catalog, ClearCollect(CollectEdificios, Filter(..., Status_ABMUnid="Alta")); used to populate edificio/habitación combos (fields field_1=room, field_2=building, Ventilacion_ABMUnid flag, Frecuencia_ABMUnid) and as target of LookUp() when patching
- '99.ABM_Frecuencias' — ClearCollect(CollectFrecuencias, Filter(..., Status_FE="Activo")); Distinct(Frecuencia_FE) feeds the frequency combos
- '00.Usuarios' — ClearCollect(CollectTecnicos, Filter(..., Perfil_Usr="Tecnico")); Distinct(ConcatName_Usr) filtered again by Status_Usr="ALTA" feeds the técnico combo in Assign modal; also LookUp by Edificio_Usr for default technician
- CollectPermisos (in-memory, loaded elsewhere in the app) — Filter by *_LPP="SI" columns keyed to VarUser.Perfil_Usr, drives the left-nav item list; not a direct SharePoint read on this screen but a dependency
- 'Documentos' (SharePoint doc library) — referenced only inside a commented-out block (img_details_GVE OnSelect) to fetch photos attached under 'Documentos compartidos/Ventilaciones/{ID}/'; not actually executed in the current build

## role_logic
Left-nav items are filtered per VarUser.Perfil_Usr against a permissions collection (CollectPermisos, one Admin_LPP/Operador_LPP/Tecnico_LPP/Recepcion_LPP='SI' column per profile) — Compras profile gets an empty nav (Blank()), implying Compras users effectively can't navigate anywhere from here (likely landing on a different home screen entirely). On this screen specifically, two hard financial-impact actions are role-gated by a direct string check rather than the permissions collection: 'Agregar Edificio/Ventilación' button and the row delete icon both require VarUser.Perfil_Usr = "Admin" Or "Supervisor Ventilaciones" (plus, for delete, Estado_VE must still be "Pendiente"). The Assign and quick-assign-icon logic also references that same Admin/'Supervisor Ventilaciones' pair in a comment, but the live Visible property was hardcoded to false, so that gate is currently moot (nobody can reach Assign from the grid).

## visual_notes
Color language for the 5 status pills (all 24-25px rounded pill icons with tinted fill/stroke + matching dark text, exact hex from the SVGs): Pendiente = fill #FFE7C2 / stroke #FEC877 / text #C07402 (orange, 'warning' tone); Asignada = fill #FFFDD8 / stroke #F0EB69 / text #807B07 (yellow); Programada = fill #DBFFCA / stroke #CBEABD / text #3A8517 (green); Realizada = fill #C5DCFF / stroke #A0C1F2 / text #043A87 (blue); Eliminada = fill #FFC9C7 / stroke #E6ABA8 / text #A6100A (red). Row action icons are simple 24x24 line icons: person silhouette (assign), warning triangle (incident flag, stroke #C03E02), eye (details, stroke #424E66), trash outline (delete, stroke #A6413B). Overall app shell uses a dark navy frame (#23313E) around a white content card, matching the rest of the desktop app's shared sidebar/header pattern — this chrome is identical across screens and should become one persistent AppShell/Sidebar layout component rather than being reimplemented per screen. All 6 popups follow one consistent recipe: full 1366x768 dark scrim (black 12% opacity, some with a 1-2px backdrop blur) behind a white rounded-corner (12px) card of varying size positioned at a fixed X/Y, with a dark 'primary' pill-shaped button (#232323/#19222B fill) for the confirming action and ghost/transparent buttons for cancel — this maps cleanly to one reusable centered-Modal recipe with size variants, not six bespoke overlays. Empty-state illustration (002.svg) is a white rounded card with a light-blue circular icon backdrop, used generically as a 'no results' placeholder.

## react_mapping
Treat the left-nav + header chrome as the shared Sidebar layout primitive (logo, role-filtered vertical nav list, logout) wrapping every screen — do not rebuild it here. Page body: Table (desktop) with a StatusBadge column (5 fixed variants mapped 1:1 to the hex pairs above — Pendiente/Asignada/Programada/Realizada/Eliminada), columns Edificio, Habitación, Última limpieza, Próxima/Programada, and a trailing action-icons cell (delete icon rendered conditionally via role+status; omit or feature-flag the Assign and Details row-icons since they're dead in the current app and product should decide whether to resurrect them). Toolbar row: Input with search icon (debounced client filter), an icon Button for refresh, a primary Button 'Agregar' (disabled via DisplayMode logic → React disabled prop driven by role check), and a filter trigger Button opening a small Popover/dropdown panel (not a full Modal — it's anchored, non-blocking) containing three multi-select Combobox fields (Mes/Año, Estado, Edificio) plus an Apply button. Empty state: reuse/extend the kit's EmptyState pattern for when the filtered list is empty. The 6 full-screen overlay popups all collapse into ONE Modal recipe (centered, scrim + rounded white card, size prop for small/confirm vs. large/form variants) reused for: Create Ventilación (form: Select edificio → Select habitación (options depend on edificio) → Select frecuencia → DatePicker → Save/Cancel, Save disabled until all fields set), Delete confirm (small danger Modal, confirm/cancel), Logout confirm (small Modal, confirm/cancel), and — if reinstated — Assign (form: DatePicker + Select técnico + conditional Select frecuencia when incident) and View-detail (read-only note + optional image, could alternately become a right Drawer since it's read-only content rather than a form). Loading overlay maps to the kit's full-page Spinner/backdrop shown while any list/patch mutation is in flight (React Query isLoading/isMutating, replacing the manual LoadingLocal boolean). No Toast component is needed for this screen specifically since the legacy Power Apps version never called Notify() — but the React port should still add success/error Toasts on the three Patch operations (create/assign/delete) since silent failures were a UX gap in the original. For mobile: convert the Table to the standard Card-list (already visually a two-line card per row in Power Apps, so this is a natural fit), turn the filter Popover into a bottom-sheet, and turn the Create/Assign Modals into full-screen bottom-sheets per the kit's mobile form pattern.
